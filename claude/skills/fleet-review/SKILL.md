# fleet-review — 並列マルチエージェントレビュー

`agy` / `codex` / `opencode` の3系統コーディングエージェントを **同時起動**して
同じ対象をレビューさせ、その出力を **1本の統合レポート**に取りまとめる。

対照的な backend を並べることが目的。3者が独立して指摘し、**複数が一致した
指摘＝高信頼**、**単独指摘＝要確認**として重み付けできる。専用エージェントは
`~/.claude/agents/`（`agy-reviewer` / `codex-reviewer` / `opencode-reviewer`）に
設定済みで、各自が diff 取得からツール実行・整形まで自律的に行う。**呼び出し側
（あなた）はファイルを読まない** — コンテキスト収集はエージェント側の責務。

## 引数の解釈（`$ARGUMENTS`）

`$ARGUMENTS` から **対象 (target)** と **レビューモード** を読み取る。省略された
要素は既定値を使う。

- **target**（第1引数、任意）:
  - 省略時は **カレントプロジェクト**（cwd）。エージェントはローカルなら直接、
    リモート target 名なら `dev run <target>` 経由で実行する。
  - target 名か不明なら `dev targets --json` で確認する（推測しない）。
- **モード**（キーワード、任意。既定 = `diff`）:
  - `diff` / 「変更」/「uncommitted」→ 未コミット変更（`git diff HEAD`）
  - `staged` → ステージ済み（`git diff --staged`）
  - `base <branch>` / 「PR」/「ブランチ」→ ブランチ差分（`git diff <branch>`）
  - `full` / 「全体」/「codebase」→ リポジトリ全体

引数が空なら「カレントプロジェクトの未コミット変更」を既定とする。

## 手順

### 1. 対象の確認（軽量）

- target が明示されていて不明なら `dev targets --json` で存在を確認。
- レビュー対象が本当にあるか軽く確認する（例: cwd で `git status --short` /
  `git diff --stat`）。**diff が空**なら、その場でユーザーに知らせて停止し、
  無駄な3エージェント起動をしない。full モードでは省略可。

### 2. 3エージェントを並列起動（1メッセージで同時に）

**Agent ツールを1つのメッセージ内で3回**呼び、`agy-reviewer` /
`codex-reviewer` / `opencode-reviewer` を同時に起動する（直列に待たない）。
各プロンプトには **同一の対象・モード**を渡し、自己完結させる。テンプレート:

```
対象: <target 省略時は「カレントプロジェクト（cwd）」と明記>
モード: <diff HEAD | staged | base <branch> | full>

上記の対象を <モード> でレビューせよ。コンテキスト（diff / ファイル）は
すべて自分で収集すること。指摘は「重大度」「ファイル:行」「理由」を
必ず添えて、あなたの標準出力フォーマットで返せ。

# 任意: 変更の意図と着目点が既に分かっている場合のみ添える（フォーカスブリーフ）
変更の意図: <1-2 文。何をどう変えたか>
重点確認: <壊れやすい不変条件を疑問形で 2-4 個。
  例「空の sort_ts は同ランク他種別の並びを乱さないか？」
     「key は選択復元に使うので不変のままか？」「更新漏れの構築サイトは無いか？」>
ただし上記はヒント。このフレーミングの外にある問題も必ず報告し、指摘は
自分で source を裏取りすること（こちらの枠組みを鵜呑みにしない）。
```

- リモート target の場合は「`dev run <target>` 経由で実行せよ」と明記。
- `base` モードなら `<branch>` を具体名で埋める（不明なら既定ブランチを確認）。
- **編集はさせない**: これはレビューのみ。各プロンプト末尾に
  「This is review only. Do NOT edit, create, or delete any files.」を付ける。
- **フォーカスブリーフは任意で、命中率を大きく上げる**。あなたが変更内容を
  把握している（自分で書いた／ユーザーが説明した）ときだけ添える。壊れやすい
  不変条件を**疑問形で名指し**すると各 reviewer がそこを重点検査する。ただし
  **独立監査**（自分が書いていない・素の目で見てほしい）目的なら、ブリーフは
  「意図」1 文に留めて誘導を避ける。どの場合も「フレーミング外の問題も報告せよ」
  の一文は必ず残し、確証バイアスで独自の発見を潰さないこと。

### 3. 取りまとめ（統合レポート）

3者の出力が揃ったら、あなた自身が突き合わせて **1本のレポート**に統合する。
生の3出力をそのまま貼らない。次を行う:

- **名寄せ・重複排除**: 同じファイル:行/同じ問題を1件に束ね、どのレビューアが
  指摘したかを併記する。
- **合意による重み付け**: 2〜3者が一致した指摘を最上位（高信頼）に。単独指摘は
  「要確認」として分けて扱う。
- **合意の独立性を確認**: 一致でも、複数の reviewer が CLI 脱落で自前推論
  （下記「失敗時の扱い」）にフォールバックしていたら、それは*別 backend の独立
  一致ではない* — 同じ Claude が 3 回言っただけになる。高信頼へ上げるのは
  **実際に別 backend が走った上での一致**のみ。フォールバック同士の一致は
  「要確認」寄りに落とす。
- **重大度で並べる**: Critical / High → Medium → Low/Nits の順。
- **矛盾の明示**: 指摘が食い違う場合は両論を残し、あなたの判断を1行添える。

出力フォーマット:

```
# Fleet Review — <target> (<モード>)

対象: <何をレビューしたか / diff の範囲・規模>
レビューア: agy / codex / opencode（CLI 実走 N/3, フォールバック M）

## 🔴 高信頼（複数一致）
- [Critical] <file>:<line> — <問題>（agy, codex）
  → <推奨対応>

## 🟡 個別指摘（単独・要確認）
- [Medium] <file>:<line> — <問題>（opencode のみ）

## 🟢 軽微 / Nits
- ...

## まとめ
<全体評価。マージ可否の所感、最優先で潰すべき点 top3>
```

## 失敗時の扱い

- あるエージェントがエラー・認証失敗・タイムアウトした場合、**残りの結果で
  レポートを完成させ**、どのレビューアが落ちたかと理由をレポート冒頭
  （「CLI 実走 N/3」）に明記する。全滅時のみ、原因を報告して停止する。
- **レイテンシ**: 1 reviewer あたり実測 3〜6 分かかる（特に `agy` が最遅）。
  遅さを失敗と誤認して途中で打ち切らない。3 者を並列起動しているので、全体の
  待ち時間は最遅の 1 者ぶんで済む。
- **バックボーン脱落の検知**（今回の agy で実際に発生）: reviewer の CLI が
  タイムアウトや出力切り詰めで実質使えず、エージェントが**自前推論（Claude +
  Bash で source を直接確認）でレビューを埋めた**場合、それは「別 backend の
  独立指摘」ではなく **Claude 自身のレビュー**になっている。この skill の価値は
  *対照的な backend を並べる*ことなので、この脱落は信頼度に直結する。reviewer
  出力に「CLI がタイムアウトしたので source を直接確認した」等の記述があれば、
  その reviewer は**フォールバック**として数え（ヘッダの M に計上）、その指摘は
  backend 由来ではなく Claude 推論として統合時に区別する（上記「合意の独立性」）。
- `opencode` はセキュリティ上フリーモデル禁止（`--model` 明示必須）。これは
  `opencode-reviewer` 側で処理されるので、こちらで指定する必要はない。

## 注意

- **二重オーケストレーション禁止**: このスキルは3レビューア以外に別 agent を
  起動しない。自分でできる名寄せ・統合を他 agent に投げない。
- task に統合されたレビュー（結果を task artifact に保存し phase 遷移させる）が
  欲しい場合は、このスキルではなく `dev task review <T-id>` を使う。
